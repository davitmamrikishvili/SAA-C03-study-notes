# KMS - Key Management Service

> [!INFO] Definition
> AWS Key Management Service (KMS) is a managed service that makes it easy for you to create and control the cryptographic keys used to protect your data.

## Core Characteristics
* **Public & Regional Service**: Accessible via public endpoints but restricted to a specific region.
* **Storage**: Keys never leave KMS.
* **Compliance**: Provides <span style="color:rgb(255, 192, 0)">FIPS 140-2 (Level 2)</span> security.
* **Functionality**: Performs cryptographic operations (encrypt, decrypt, re-encrypt, etc.) and manages both symmetric and asymmetric keys.

---

## CMK - Customer Master Keys
The "Primary" key managed by KMS. It is a logical container for the actual physical key material.

* **Components**: ID, Creation Date, Policy, Description, and State.
* **Material**: Physical key material can be generated by AWS or imported.
* **Size Limit**: CMKs can directly encrypt/decrypt up to <span style="color:rgb(255, 192, 0)">4 KB</span> of data.
* **Security**: The CMK is always stored encrypted within KMS.

### Types of CMKs
1.  **AWS Managed**: Created automatically when you use an AWS service that needs KMS (e.g., S3). Rotated every **1,095 days** (3 years).
2.  **Customer Managed**: Created explicitly by you. Highly configurable. Rotated **once a year** (if enabled).

### Key Facts
* **Rotation**: CMKs support rotation. The CMK object contains the current and all previous backing keys.
* **Aliases**: You can create friendly names (aliases) for keys; these are regional.
* **Region Lock**: CMKs are isolated to the region where they were created.

---

## DEK - Data Encryption Keys
Used for Envelop Encryption (encrypting data larger than <span style="color:rgb(255, 192, 0)">4 KB</span>).

* **Generation**: Generated by KMS using a CMK via the <span style="color:rgb(240, 75, 200)">GenerateDataKey</span> API call.
* **Management**: KMS **does not store** DEKs and does not perform encryption/decryption with them. Your application or the AWS service manages the DEK.
* **The Two Versions**: When generated, KMS returns:
	1.  **Plaintext DEK**: Used immediately to encrypt the data.
	2.  **Ciphertext DEK**: An encrypted version of the DEK (encrypted by the CMK).

### Envelope Encryption Process
**Encryption:**
1.  Ask KMS to generate a DEK using a CMK.
2.  Use the **Plaintext DEK** to encrypt your data.
3.  **Discard** the Plaintext DEK from memory.
4.  Store the **Encrypted Data** side-by-side with the **Ciphertext DEK**.

**Decryption:**
1.  Send the **Ciphertext DEK** to KMS.
2.  KMS decrypts it with the original CMK and returns the **Plaintext DEK**.
3.  Use the Plaintext DEK to decrypt your data.
4.  **Discard** the Plaintext DEK.

---

## Key Policies & Security
* **Resource Policy**: Key policies are a type of resource-based policy. Every CMK **must** have one.
* **Default Trust**: KMS must be explicitly told to trust the AWS Account (via the Root user) to allow IAM policies to manage access to the key.
